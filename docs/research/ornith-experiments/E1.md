# E1 — minecraft-monitoring: LogQL log-alert migration to the Loki ruler

**Date:** 2026-07-16 · **Target:** github.com/evince55/minecraft-monitoring · **PR:** evince55/minecraft-monitoring#5
**Harness:** Arm B (Ornith-35B authors the fix; Claude scopes + verifies) · **Model:** ornith-1.0-35b-Q4_K_M @ llama.cpp (Windows/Tailscale, `<tailscale-ip>:8080`)

## What this experiment measures

Can a locally-hosted 35B model do real build work — not answer a single-shot
probe, but produce a correct, deployable fix to an existing project? This is
the "agentic-probe" successor from the R1 roadmap, run in the cheaper Arm B
mode (Claude orchestrates; Ornith generates all fix code).

## The bug (found by Claude in survey; promtool-proven)

`manifests/alerting/log-alerts.yaml` was a `kind: PrometheusRule` whose four
rules use **LogQL**, e.g. `count_over_time({app="minecraft"} |= "ERROR" [5m])`.
Prometheus evaluates only PromQL; `promtool check rules` rejects all four:
`parse error: unexpected character: '|'`. The rule group never loads → the log
alerts silently never fire. The repo's CI missed it: kubeconform checks only
k8s object schema, not rule expressions, and promtool isn't in the pipeline.

(A second candidate — a ServiceMonitor `jobLabel` pointing at a nonexistent
label — was investigated and **dropped**: it resolves via the operator's
service-name fallback. Verify before asserting applies to survey findings too.)

## Ornith's build — two turns

**Turn 1** (73s, 1412 tok): correct *strategy* on the first try — recognized
the rules belong on the **Loki ruler**, proposed a ConfigMap + values change +
deleting the PrometheusRule. But three repo-blind implementation errors:
1. **Wrong chart API / nesting** — proposed a top-level `ruler:` block; this is
   a *wrapper* chart (its own header says subchart config nests as `loki.loki.`),
   and the grafana/loki chart has no such `ruler.enabled/configPath` values.
2. **Wrong Alertmanager address** — guessed `alertmanager.monitoring:9093`.
3. **Rewrote the alert expressions** — dropped the original
   `sum by (instance)(...) / 300` rate normalization, changed `for:` durations
   and match strings (`"left the game"` → `"disconnect"`, `"Chunk"|="load"` →
   `"chunk"`), i.e. a loose re-creation, not a migration.

**Turn 2** (32s, 787 tok) — re-prompted with three *verified* repo facts
(wrapper-chart double-nesting; real service name
`kube-prometheus-stack-alertmanager.monitoring:9093`; the four original
expressions verbatim). Ornith fixed all three fed issues: expressions restored
exactly, correct AM URL, correct sidecar label `loki_rule=1`. One residual
error: it nested `sidecar` under `loki.loki.` (subchart structured config) when
it is a top-level chart value (`loki.sidecar`).

## Claude interventions (Arm B)

1. One re-prompt with verified repo facts (allowed by the rubric).
2. One manual nesting fix: `sidecar` moved `loki.loki.sidecar` → `loki.sidecar`
   (confirmed against the extracted subchart's `values.yaml`: `rulerConfig` at
   subchart-`loki:` level, `sidecar` at top level), plus rule-path alignment
   (`storage.local.directory` = sidecar `folder` = `/rules`).
3. Mechanical consistency: `verify.sh` updated to check the new ConfigMap;
   confirmed no other dangling references to the deleted PrometheusRule.

converged to a correct, deployable result after one fact-injection re-prompt and

Ornith the repo directly should close most of this gap — the ideal E1 A/B contrast.
