# E4 — homelab-status: a stdlib status page from scratch

**Date:** 2026-07-16 · **Target:** new mini-project → github.com/evince55/minecraft-monitoring#6 (`homelab-status/`)
**Harness:** Arm B (Ornith-35B authors; Claude scopes + verifies) · **Model:** ornith-1.0-35b-Q4_K_M @ llama.cpp (Windows/Tailscale)

## What this experiment measures

E1 tested Ornith on a *bug fix in an existing project*. E4 tests the opposite:
**a from-scratch build** — a whole small mini-project — where there is no
existing code to ground on, only a spec. Does the weakness shift from
repo-grounding (E1) to something else?

## The task

Build a dependency-free "homelab status page": a stdlib-only Python server that
polls configured service-health endpoints server-side (no browser CORS), caches
results, and serves both an HTML dashboard and a `/api/status` JSON endpoint.
Seeded from the owner's real services (aria-backend, the minecraft stack's
Prometheus/Grafana/Loki/incident-responder).

## Ornith's build — two turns

**Turn 1** (183s, hit the 5000-token cap): a genuinely well-structured server —
correct architecture (background daemon poller, lock-guarded cache, per-check
timeout + try/except so one dead endpoint can't crash others), both routes,
external `config.json`, a clean dark-theme dashboard with client-side refresh.
But two defects:
1. **Fatal, language-level:** the entire HTML/CSS/JS page was built with a
   Python **f-string**, while the CSS/JS contain hundreds of literal `{`/`}`.
   Python can't compile it — `SyntaxError: f-string: expecting a valid
   expression after '{'` at the JS `try {`. The program never runs.
2. **Correctness:** the Grafana check expected `message: OK`, but Grafana's
   `/api/health` returns `database: ok` — the check would report Grafana down
   when healthy.
3. The README was truncated by the token cap.

**Turn 2** (186s) — re-prompted with the exact compiler error + root cause
(f-string literal braces) and the Grafana field fix, with the instruction to
restructure to a plain template + `%%TOKEN%%` `.replace()` substitution rather
than doubling every brace. Ornith did exactly that: compiles cleanly, uses the
placeholder-token approach, Grafana field corrected.

## Claude interventions (Arm B)

1. One re-prompt (the fatal f-string bug + Grafana field).
2. Wrote the README (truncated by the token cap both turns) — docs scaffolding.
3. Authored a mock health endpoint + test config to exercise up/down paths.

## Verification (end-to-end — the point of Arm B)

- `py_compile`: clean after turn 2.
- **Ran it** against a mock healthy endpoint (`{"status":"ok"}`) and an
  unreachable one. `/api/status` correctly returned `up` (200, latency) for the
  healthy checks and `down` with `[Errno 61] Connection refused` for the dead
  one — per-service isolation confirmed.
- **Rendered the dashboard** in a browser: red "1 service(s) down" banner, two
  green UP cards with latency/HTTP-code, one red DOWN card showing the
  connection error. (Captured during verification; see minecraft-monitoring#6.)

## Grade: **PARTIAL** (strong, recovered)

The turn-1 defect is the interesting result. In E1 (bug fix) the errors were
**repo-grounding** (chart wiring, service names). In E4 (from-scratch) there was
no repo to be blind to — and the fatal error was instead a **language-level
generation bug**: a huge literal template stuffed into an f-string. Same
altitude of finding as E1 (right structure, wrong low-level detail), but the
failure mode moved from *environment knowledge* to *language mechanics*. Both
were caught only by actually running the code — reinforcing that Ornith output
must be executed, not eyeballed. After one precise re-prompt it produced a
correct, genuinely working artifact.

Cross-experiment pattern (E1 + E4): **Ornith-35B reliably gets architecture and
structure right, and reliably needs one verification-driven correction on a
concrete detail** — repo-specific in a codebase, language-specific from scratch.
For the routing model: Ornith is a strong *local generator* under a verifying
orchestrator, not yet a fire-and-forget builder. The Arm A rerun (opencode +
tools, letting Ornith run the code itself) is the test of whether a self-verify
loop closes that gap without a human/Claude in the middle.
